## Problem

The original intent (Spec 9 Step 6b, Spec 16) was that language plugins provide
their own tree-sitter grammar as a WASM artifact, loaded by the engine **at
runtime** â€” so a platform-specific engine binary can be extended with new
languages (grammar + analyzer + IIR lift) without recompiling.

That was never wired. Today:

- All parsing is done by **CGO built-in grammars** (`smacker/go-tree-sitter`,
  `internal/indexer/parser/grammar.go` `registerBuiltIns`).
- `loadWASMGrammar` is a **stub** that errors on every call (has since the file's
  first commit). No grammar WASM artifacts exist in the repo.
- Because tree-sitter is a CGO dependency, the engine is `CGO_ENABLED=1` and the
  release cross-compiles the C **natively per architecture** (6Ă—) from one
  `ubuntu-latest` runner â€” which is the only reason we need `zig` (a CGO
  cross-compiler, **not** a Windows dependency).

## Decision

Adopt **Path A**: run tree-sitter **as WASM on wazero** (the pure-Go runtime the
engine already uses for plugin logic). This makes grammars runtime-loadable and,
as a direct consequence, lets the engine drop CGO.

One lever, three payoffs:
1. Grammars become runtime-loadable WASM plugins (the original intent).
2. `CGO_ENABLED=0` â€” tree-sitter no longer compiled per-arch.
3. `go build` replaces the entire zig/clang/osxcross cross-compile apparatus;
   Windows returns for free once CGO is gone.

### Key architectural facts

- The **tree-sitter core** is compiled to WASM **once**. `wasm32` bytecode is
  architecture-independent, so the *same bytes* are embedded and reused by every
  platform binary. Built once, not per build.
- A **grammar** is a separate small WASM **side module** (emscripten/LLVM
  `dylink.0` format) containing only that language's parse tables (+ optional
  external scanner). It shares the core's linear memory and function table.
- **Core vs grammar sizes:** core ~100KBâ€“fewĂ—100KB (shipped once); grammar
  ~tens KB to a few MB (TypeScript/C++ are the fat outliers). Ten languages =
  one runtime + ten small grammars, not ten runtimes.

### Artifact strategy â€” Option 1 (vendor in-repo)

The core + default grammar WASMs are **committed into the engine repo** and
embedded via `//go:embed`, regenerated by a **dedicated version-bump / manual
-dispatch pipeline** (not per-commit). The pipeline records provenance
(tree-sitter source commit, toolchain version, sha256). Rationale: keeps
`go build` hermetic/offline/reproducible, reuses the existing `//go:embed
internal/indexer/defaults` mechanism, and avoids another opaque
"sourced-elsewhere" binary surface (which already caused confusion with the
phantom `*-grammar.wasm` names in `defaults.go`'s skip-if-absent list). The
WASM-building toolchain is thereby **quarantined** to one rarely-run workflow;
the engine build itself stays toolchain-free.

## How runtime loading works (the dynamic linker)

The core is a **MAIN module** (non-PIC); grammars are **PIC SIDE modules**
sharing the main's memory + `__indirect_function_table` (emscripten
`MAIN_MODULE=2` model). A grammar side module imports from `env`:
`memory`, `__indirect_function_table`, and the globals `__memory_base` /
`__table_base` (0 function imports for scanner-less grammars like Go).

Per-grammar load sequence:
1. Parse the side module's `dylink.0` custom section â†’ `mem_size`, `mem_align`,
   `table_size`, `table_align`.
2. `memory_base = malloc(mem_size)` (rounded up to `mem_align`), via the core's
   exported `malloc`.
3. `table_base = dl_table_grow(table_size)` â€” grow the shared indirect function
   table, returning the previous size.
4. Set the core's exported **mutable** globals `__memory_base` / `__table_base`.
5. Instantiate the side module; wazero resolves its `env.*` imports against the
   core module (instantiated with name `env`). Its active element/data segments
   self-install functions/data at the bases.
6. Call the side's `tree_sitter_<lang>()` â†’ `TSLanguage*` (a shared-memory
   offset), hand to `ts_parser_set_language`, parse.

### Two constraints discovered in the POC

- **wazero exposes no host-side table API** (`api/wasm.go: // TODO: Table`). The
  host cannot grow/populate the shared table directly. Workaround: a
  `table.grow` helper compiled/embedded so it runs *inside* wasm (host calls the
  export). Only *growing* is host-driven; the side module self-installs its
  entries via its own element segment.
- **`table.grow` needs the wasm `reference-types` feature**, which `zig cc`
  (0.13) refuses to propagate to its integrated assembler. Resolutions:
  (a) build with wasi-sdk clang (allows `-mattr=+reference-types`); or
  (b) **hand-emit the ~40-byte table-grow helper module in Go** (toolchain-free)
  and enable reference-types in wazero's `RuntimeConfig`. POC uses (b).

## Toolchain notes (POC-proven with zig 0.13)

- Core (MAIN, non-PIC): `zig cc --target=wasm32-wasi-musl -mexec-model=reactor
  src/lib.c -Wl,--export-table -Wl,--export=malloc â€¦ -Wl,--export=ts_*`.
- Grammar (PIC SIDE): compile `parser.c -fPIC`, then `zig wasm-ld
  --experimental-pic -shared --export=tree_sitter_<lang> --allow-undefined`.
  Produces a real `dylink.0` side module, structurally identical to the
  `tree-sitter-wasms` (emscripten) artifacts.
- Mutable base globals `__memory_base`/`__table_base`: exported from the core
  (mutable-globals is baseline; does not need reference-types).
- `dl_table_grow`: hand-emitted helper (path b) â€” reference-types.
